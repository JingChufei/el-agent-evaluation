# Agent 评测维度判定契约 · 模板规范

> **本文档定义"判定契约"的 12 项标准结构。**
> 它是规范层文档，独立于任何具体业务。任何新增评测维度都需要按本模板填写一份契约，作为该维度的标注规范、实现规范与校准规范的统一来源。
>
> **配套文档**:`evaluation-contracts-v{X.Y}.md`(实例层,绑定当前业务,按本模板填写的全部维度契约集合)
>
> **版本**: v1.0 / 2026-05-29

---

## 一、这份文档给谁用,用来干什么

判定契约的读者至少包含四类角色,模板设计满足全部四类的最小需求:

| 角色 | 关心的子项 | 用契约做什么 |
|---|---|---|
| 评测工程师 | ④⑥⑦⑧⑨ | 实现 checker 代码,确定输入输出 |
| 标注人员 | ⑤⑨ | 产出 ground-truth,理解标注规范 |
| Judge prompt 设计者 | ②⑤⑦⑧⑩ | 写 rubric,设计 Judge 输入 |
| Bad case 复核者 | ②⑧⑨⑪ | 判断 case 是否被冤枉或漏判 |

**判断子项是否必要的标准**:如果某个子项对上述四类读者中任何一类都不构成行动指引,它就是冗余的;反过来,如果某类读者需要的关键信息找不到对应子项,模板就是缺失的。

---

## 二、设计原则

下列原则贯穿全部 12 项,理解原则比照搬模板更重要。

**原则 1 · 契约是对外接口,不是对内实现。** 判定规则用伪代码或决策树,不绑定具体语言/框架。换实现不应触发契约修订。

**原则 2 · 每一项都要能反向追溯。** 读 ⑥ 输出指标应能找到 ⑧ 判定规则里对应的计算;读 ⑪ 失败模式应能找到 ⑧ 或 ⑥ 中对应的防御设计。如果某项内容找不到下游对应,它就是装饰性的。

**原则 3 · "什么不在这里"也要明确。** 维度间的边界纠纷大多源于契约没写清"不管什么"。⑨ 边界条件应显式列出"哪些 case 不走此维度""哪些信号不归此维度统计"。

**原则 4 · 契约本身要可校准。** ⑩ 不只是"我希望达到 κ>0.6",而是"如何抽样、如何统计、不达标怎么办"。这是评测可信落地的硬抓手。

**原则 5 · 写在契约里的失败模式不能后续被删。** ⑪ 失败模式与防 hacking 设计是契约的"防御性条款"。后人迭代规则时容易把原本是反 hacking 设计的对照指标误删,⑪ 必须长期保留并随每次修订更新。

## 三、12 项子项详解

### 组 1 · 身份与定位

#### ① 维度 ID 与名称

**用途**:为该维度提供稳定的索引锚点。Bad case 库、评测报告、CI 输出、回归监控都按维度 ID 切分。

**强制要求**:
- ID 形如 `D{N}_{SHORT_NAME}`,全大写,下划线分隔,如 `D4_TRAJECTORY_MATCH`
- ID 一旦冻结,不允许修改(改名会让历史数据无法对齐)
- 同时提供中文名 + 英文名

**写法示例**:
```
ID:   D4_TRAJECTORY_MATCH
中文名: 调用链匹配
英文名: Trajectory Matching
```

#### ② 评测对象

**用途**:用一句话说清"这维度测什么"。

**强制要求**:严格一句话。如果一句话说不清,说明维度本身定义不内聚,应拆维度。

**写法示例**(良好):
> Agent 实际调用序列与 gold chain 的吻合程度,同时覆盖过程正确性、错误恢复、合理替代路径。

**写法示例**(不良 — 引入了多件事):
> 测试 Agent 调用工具是否合理,顺序对不对,有没有正确处理错误,以及最终结果是否符合预期。

后者应该拆成多个维度。

#### ③ 所属层级 + 过程/结果定位

**用途**:决定该维度在 pipeline 中的执行顺序与报告聚合方式。

**强制要求**:
- 层级: `结果层 | 过程层·宏观 | 过程层·微观 | 质量层 | 系统层`
- 过程/结果: `过程式 | 结果式 | 混合`

按层和按过程/结果切片,是报告中比按维度逐条罗列更清晰的聚合方式。

---

### 组 2 · 输入输出

#### ④ 评测输入

**用途**:明确该维度需要从 trajectory 取哪些字段。

**强制要求**:列出每个字段的具体路径,例如 `trajectory.tool_calls[].name`,不写"trajectory 中相关信息"。

**关键意义**:此项直接对应 trajectory schema 设计。schema 未准备好的字段意味着对应维度无法实现,这种依赖必须显式可见,否则到实施阶段才发现 schema 缺字段就是返工。

#### ⑤ Ground-truth 需求

**用途**:让标注人员知道要标什么、怎么标。

**强制要求**:
- 明确是否需要 per-case 标注: `需要 | 不需要 | 部分需要`
- 如需要,给出 YAML schema 示例(可直接当标注规范用)
- 如不需要,说明依赖什么(工具注册表、共享词表、trajectory schema 字段等)

#### ⑥ 输出指标

**用途**:该维度产出哪些数字,分 case 级和聚合级。

**强制要求**:
- case 级指标(每条 case 产出什么)
- 聚合级指标(全集报告什么)
- 必要的细分维度(按错误类型、按意图类别、按 case_type 等)

显式列出细分指标至关重要——不写出来,工程师只会出一个总数,失去诊断价值。

---

### 组 3 · 判定逻辑

#### ⑦ 实现方法

**用途**:说明该维度采用哪种或哪些评测方法。

**强制要求**:不是单选标签,而是"主 + 备"结构。
- 主路径: 规则 / LLM-Judge / 人工 / 状态执行
- 兜底: 当主路径无法判定时退化到什么

**关键意义**:这个结构防止团队为图省事跳过规则直接上 Judge。例如"主路径=规则,兜底=人工抽检"明确告诉工程师不该把规则可以判定的部分外包给 Judge。

#### ⑧ 判定规则

**用途**:契约里最重的一项,也是最容易写成"自然语言描述"的一项。

**强制要求**:
- **用伪代码或决策树写**,不要自然语言
- 自然语言再清晰都会有歧义,工程师实现时会塞进自己的假设
- 多粒度/多分支的判定要分块写,每块独立可读

**写法示例**(良好):
```
IF any(action matches forbidden_patterns for action in trajectory):
    RETURN score=0, reason="forbidden_triggered"

hits = count(req satisfied for req in required_items)
total = len(required_items)
IF hits == total: RETURN pass=1
ELSE: RETURN pass=0, score=hits/total
```

**写法示例**(不良):
> 如果触发了禁止操作就判 0 分,否则看必要项满足多少,全部满足才算通过,否则按比例打分。

#### ⑨ 边界条件与例外

**用途**:明确"什么情况下这条规则不适用"。

**强制要求**:显式列举:
- 哪些 case 不走此维度
- 哪些信号不归此维度统计
- 与其他维度的边界(防止重复计算或互相打架)
- 参数选取的合理范围与原因

**关键意义**:Bad case 复核者最常翻这一项。例外条款写清楚能省掉大量"这 case 该不该算"的争论。

---

### 组 4 · 质量与边界

#### ⑩ 校准要求

**用途**:量化该维度的可信度门槛。

**强制要求**:
- **规则维度**: 写"人工抽检方法 + 精确率/召回率 + 验收阈值"
- **Judge 维度**: 写"κ 阈值 + 校准批次规模 + 不达标处理"
- **混合维度**: 分别写

**关键意义**:此项是评测可信落地的硬抓手。不达标的维度不能放行进入全量评测,防止"先实现再说"的工程惯性。

#### ⑪ 已知失败模式与防 hacking 设计

**用途**:每个维度都有可能被模型钻空子或被规则误判,⑪ 显式记录这些坑和对应的防御。

**强制要求**:
- 已知的假阳性/假阴性来源
- 已知的模型规避策略(如"无脑重试""遮羞布 hedging")
- 对应的防御设计(对照指标、反向检查、归一化预处理等)

**关键意义**:此项的价值不在当下,而在半年后团队迭代规则时——没有这一项,新人很容易把原本是反 hacking 设计的对照指标误删。每次修订都应回顾并更新此项。

---

### 组 5 · 运维信息

#### ⑫ 版本与负责人

**用途**:契约本身的版本管理与责任归属。

**强制要求**:
- 版本号: 形如 `v0.1`,每次实质性修订递增
- 修订日期: ISO 格式
- 负责人: 唯一可问责的人,不是团队名

**关键意义**:评测体系不是一次写完就不动的。没有版本号就追溯不了"为什么这次跑分变了"——是模型变化还是契约变化造成的。

```markdown

**⑨ 边界条件与例外**:

## 五、版本管理建议

**契约何时更新**:
- 工具集变化(新增/废弃工具) → ④ ⑤ ⑧
- 业务策略变化(澄清规则、合规要求) → ⑤ ⑧ ⑪
- 校准结果显示失败模式 → ⑪ ⑩
- 重大重写 → 整体大版本

**修订记录**:每次修订在契约末尾加一行 `> v{X.Y}: {修订说明}`,保留历史。

**版本兼容性**:大版本(v1.0 → v2.0)修订可能破坏历史数据对比。这类修订应在报告中显式标记并提供回归验证。
