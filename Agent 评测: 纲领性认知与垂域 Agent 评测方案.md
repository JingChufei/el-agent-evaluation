# Agent 评测: 纲领性认知与垂域 Agent 评测方案

> 本文分两部分. 第一部分是 LLM Agent 评测领域的纲领性认知, 作为方法论基础; 第二部分结合具体业务, 给出垂域 Agent 测试集构造与自动化评测 pipeline 的可落地方案. 中间以"桥接"一节点明前者如何指导后者. 

---

# 第一部分 · Agent 评测的纲领性认知

## 1. 核心概念体系

整个领域的术语可以按上下位关系组织成一棵树, 根节点是 **LLM Agent Evaluation**, 下分七个维度. 理解这七维度, 基本就掌握了讨论任何评测工作的坐标系. 

**① 被测能力维度 (评什么能力)**: 工具使用 (含工具检索、工具选择、参数填充、调用执行四个子能力) 、任务规划 (任务分解、多步规划) 、推理、记忆、自我反思与纠错、指令与策略遵循. 

**② 任务结构维度 (任务长什么样)**: 交互轮次 (单轮 / 多轮) 、工具数量 (单工具 / 多工具) 、调用组织 (序列 / 组合 / 嵌套 —— 按调用间的数据依赖关系递增) 、任务视野 (短视野 / 长视野). 

**③ 评测范式 (怎么测)**: 静态评测 vs 交互/智能体评测 (环境是否会响应模型行为) 、过程式评测 vs 结果式评测 (看轨迹还是看终态). 

**④ 评测指标 (用什么数字)**: 匹配类 (精确匹配、AST 匹配) 、完成类 (Pass Rate、Win Rate、任务成功率) 、状态类 (状态匹配、里程碑达成) 、可靠性 (pass^k, 即 k 次独立运行的全成功率) 、成本类 (Token 成本、步数效率). 

**⑤ 评分机制 (谁来判对错)**: LLM-as-Judge、规则评分、执行式评分、人工评测. 

**⑥ 评测环境 (在哪测)**: API 类型 (真实 API / 模拟虚拟 API) 、用户模拟器、沙盒、状态化环境、协议 (如 MCP). 

**⑦ 评测有效性威胁 (什么会让结论失真)**: 数据污染、基准饱和、可复现性、评分器偏向、分布内 vs 分布外 (OOD). 

需要注意, 这七维度中维度 ① 和 ④ 是真正的分类层级 (is-a 关系), 而维度 ② 更接近"属性—取值"关系 (如单轮 / 多轮是互斥取值而非子类型). 混编于一棵树是为了呈现术语全貌. 

## 2. 五类评测方法与不可能三角

按"如何判定对错"这条最有区分度的轴, 可以归出五类方法. 

**匹配式 (Reference-matching)**. 
- 核心假设: 任务存在可预先写定、可枚举的标准答案, 正确性等于与之匹配. 
- 适用于单轮函数调用、参数填充、API 选择等有明确金标准的任务. 
- 优点是全自动、零成本、可无限复跑; 局限是无法处理"多条都对的路径", 进入多轮或开放式任务即失效. 
- 代表: Gorilla 的 AST sub-tree matching、BFCL v1/v2. 

**执行/状态式 (Execution & State-based)**. 
- 核心假设: 成败可通过检查执行后的世界状态或最终产物来验证, 不关心走了哪条路. 
- 适用于有状态的多轮 agentic 任务、真实环境操作、目标可程序化校验的任务. 
- 局限是每个任务都要写校验器、构建成本高, 且部分目标天然难以状态化. 
- 代表: SWE-bench (跑单元测试) 、τ-bench (检查数据库终态) 、ToolSandbox、OSWorld、MCP-Universe. 这是目前最接近"理想"的一类. 

**模型评判式 (LLM-as-Judge)**. 
- 核心假设: 一个足够强的 LLM 能在大规模、低成本下近似人类对质量的判断. 适用于没有唯一标准答案的开放式输出、需要打质量分/胜率的场景、海量样本初筛. 
- 局限关键: 评分器偏向 (对自家或相似风格有偏好) 、随机性导致波动、强评分器成本高、输出可被"取悦评分器"地操纵. 
- 代表: ToolBench 的 ToolEval、StableToolBench. 

**人工评判式 (Human Evaluation)**. 
- 核心假设: 在细腻、主观或高风险的判断上, 人是黄金标准. 适用于发布前最终验收、主观质量与安全合规审查、给自动评测器做对齐校准的真值来源. 
- 局限显而易见: 慢、贵、不可规模化、标注者间一致性难保证. 

**分步诊断式 (Process-based decomposition)**. 
- 核心假设: 整体能力可分解为可分别测量的子能力, "在哪一步失败"与"是否最终通过"同等重要. 
- 适用于研究诊断、消融实验、定位失败根因、生成能力画像. 
- 局限是需预先定义"正确过程", 可能惩罚有效的替代策略. 
- 代表: T-Eval (六子能力分项) 、ACEBench、ComplexEval. 

**不可能三角**. 把这些方法放在一起看, 会发现它们想满足的三个目标 —— **可规模化、客观不可操纵、可处理开放式任务** —— 构成一个不可能三角, 任一方法最多同时占两个角: 
- 匹配式可规模化且客观, 但处理不了开放式; 
- LLM 评判式可规模化且能评开放式, 但可被操纵; 
- 人工评判式客观且能评开放式, 但无法规模化. 

中心的"三者兼得"目前无解, 这正是验证器困境的根源. 执行/状态式是离中心最近的折中 (牺牲了"目标可程序化"之外的覆盖面与构建成本). 

此外有几对正交的 tradeoff 会叠加: 真实 ↔ 可复现、成本 ↔ 信噪比 (pass^k 跑越多越准但越贵) 、过程效度 ↔ 结果效度、静态 ↔ 交互. 实践中没有单一方法够用, 严肃评测几乎都是分层组合: 执行/状态式做主判、LLM-Judge 兜底开放式部分、小规模人工校准评分器、外包一层 pass^k 度量可靠性. 

## 3. 未解决的核心问题

**验证器困境 (scalable oversight)** —— 最根本. 对开放式长程任务, "对错"没有可枚举的标准答案. 执行式需手写校验器, 而写校验器往往和完成任务一样难, 且会同时犯 false negative (valid 替代终态被判错) 和 reward hacking (取巧达成终态被判对). LLM-Judge 则有致命上限: 无法用一个比被测 Agent 更弱的裁判可靠地评判它. 当 Agent 逼近前沿, 裁判本身成了天花板. 

**可复现 vs 真实** —— 一对真实的不可能. 真实 API/环境会漂移 (ToolBench 的教训), 冻结/镜像换来稳定却带来失真与过时 (StableToolBench 的取舍), 选真实在线环境又背上不稳定与高成本 (MCP-Universe 的代价). 两头不能都要. 

**污染与饱和** —— 军备竞赛. 基准一旦重要就会泄漏进训练数据, "Verified"重制版只是买时间, 基准半衰期在缩短. 静态基准尤其饱和得快. 

**构念效度 / 基准—现实落差**. 我们测的是"可测的", 不是"真正重要的". 高基准分不等于真实可用. 部署中真正要紧的 —— 遇到诡异输入是否稳健、该不该停下反问、会不会做灾难性操作、长期可靠性 —— 恰恰最缺好的度量. 

**长程可靠性塌缩**. 每步 90% 成功率在 50 步任务上复合后只剩约 0.5%. pass@1 与 pass^k 的巨大落差揭示的就是这个, 但多数基准只报单一成功率, 把方差、错误恢复、误差累积全藏起来. 

**成本与安全两个长期盲区**. 多数榜单不计 token/时延/成本, "贵 10 倍只高 2 分"也算赢 (HAL 试图修正). 安全更被忽视 —— 几乎没有基准系统测试工具返回里的 prompt injection、未授权敏感操作、对抗环境下的行为. 考虑到部署风险, 这可能是最大的缺口. 

## 4. 领域主线演进

主线一句话概括: **评测的判定单位, 从"一次 API 调用对不对"一路升级到"一个长程智能体在真实环境里可不可靠"**. 

**2022–23 萌芽**: WebShop 提供最早的 sim-to-real web agent 环境, ReAct 确立"推理+行动"交错范式. 核心问题是能否让模型用工具完成任务. 

**2023 奠基**: 问题是"能不能调对". API-Bank 是第一个工具增强基准, Gorilla 带来 AST 匹配, ToolBench 用 16k 真实 API + DFSDT 把规模与复杂度拉满, AgentBench 把视角抬到"LLM as agent", GAIA 用"对人易对 AI 难"摆出天花板. 评测以单轮为主. 

**2024 标准化转折**: 三件事同时发生. 标准化 —— BFCL 把函数调用做成工业基准 (v1 AST → v3 多轮/状态式); 可复现性危机被正面解决 —— StableToolBench 实证戳破 ToolBench 的不稳定与评分器偏向, 用虚拟服务器替代真实 API; 范式升级 —— τ-bench 引入用户模拟器与 pass^k, ToolSandbox 引入有状态执行, T-Eval 拆解子能力. 年底 Anthropic 发布 MCP, 埋下伏笔. 

**2025 真实环境 / MCP 元年**: 评测重心彻底转向真实在线环境的长程智能体. MCP-Universe、MCP-Bench 等十余个 MCP 基准集中爆发, Google 在 I/O 2025 让 Gemini API 原生支持 MCP. HAL 把成本做成一等指标, τ²-bench 把用户也建模为带工具的博弈方, SWE-bench / OSWorld 出 Verified 版对抗污染. 

**2026 及未来**: 焦点落在长程可靠性、Agent 安全、互操作协议 (MCP 纵向连工具 / A2A 横向连 agent). 大厂押注方向收敛于 —— 互操作协议成为新评测底座、长程可靠性与成本被同时计入一张榜、以及谁能在不依赖更强裁判的前提下逼近可信验证 (scalable oversight). 

## 5. 地标论文锚点

按贡献谱系组织, 分前置范式 + 五条线. 

**前置范式 (评测的方法论前提)**
- **ReAct: Synergizing Reasoning and Acting in Language Models** — Yao et al., ICLR 2023, arXiv:2210.03629. 推理与行动交错的范式, 几乎所有 agent baseline 的默认骨架. 
- **Toolformer: Language Models Can Teach Themselves to Use Tools** — Schick et al., NeurIPS 2023, arXiv:2302.04761. "模型自学用工具"的开山之作. 

**一、工具调用 / 函数调用谱系**
- **API-Bank: A Comprehensive Benchmark for Tool-Augmented LLMs** — Li et al., EMNLP 2023, arXiv:2304.08244. 第一个系统性的工具增强 LLM 基准, 确立"规划/检索/调用"分阶段评测. 
- **Gorilla: Large Language Model Connected with Massive APIs** — Patil et al., NeurIPS 2024, arXiv:2305.15334. 提出 AST sub-tree matching 检测幻觉调用, "无裁判、纯结构匹配"范式影响至今. 
- **ToolLLM: Facilitating LLMs to Master 16000+ Real-world APIs** — Qin et al., ICLR 2024 Spotlight, arXiv:2307.16789. 本系列锚点: 16k 真实 API + DFSDT 标注 + ToolEval (Pass/Win Rate). 
- **Berkeley Function Calling Leaderboard (BFCL)** — Patil, Mao et al., UC Berkeley Gorilla, 2024 起持续更新. 函数调用的事实工业标准; 是持续演进的 living leaderboard 而非冻结论文. 
- **StableToolBench: Towards Stable Large-Scale Benchmarking on Tool Learning** — Guo et al., ACL 2024 Findings, arXiv:2403.07714. 实证戳破 ToolBench 的可复现性危机, 定义"虚拟 API + Solvable Pass Rate"修复路线. 讨论评测可靠性绕不开. 

**二、通用智能体评测**
- **AgentBench: Evaluating LLMs as Agents** — Liu et al. (THUDM), ICLR 2024, arXiv:2308.03688. 8 个异构交互环境, "通用 agent 评测"代表作. 
- **GAIA: A Benchmark for General AI Assistants** — Mialon et al. (Meta/HF), 2023, arXiv:2311.12983. "对人简单、对 AI 难"的北极星基准 (人类 92% vs GPT-4+plugins 15%). 

**三、真实环境智能体 (execution-based 标杆)**
- **WebShop: Towards Scalable Real-World Web Interaction** — Yao et al., NeurIPS 2022, arXiv:2207.01206. 最早的 sim-to-real web agent 环境, 后续模板源头. 
- **WebArena: A Realistic Web Environment for Building Autonomous Agents** — Zhou et al., ICLR 2024, arXiv:2307.13854. 自托管真实网站 + 功能正确性校验, 定义 web agent 评测标准形态. 
- **SWE-bench: Can Language Models Resolve Real-World GitHub Issues?** — Jimenez et al., ICLR 2024, arXiv:2310.06770. 可能是这个时代影响力最大的 agentic 基准: 真实 issue+PR、跑单元测试判对错, execution-based 典范. 
- **OSWorld: Benchmarking Multimodal Agents for Open-Ended Tasks in Real Computer Environments** — Xie et al., NeurIPS 2024, arXiv:2404.07972. 真实操作系统环境 (发布时最佳 12% vs 人类 72%), computer-use agent 地标. 

**四、对话 / 多轮智能体与可靠性**
- **τ-bench: A Benchmark for Tool-Agent-User Interaction in Real-World Domains** — Yao et al. (Sierra), 2024, arXiv:2406.12045. 引入 LLM 用户模拟器与 pass^k 可靠性指标 (后者已被 Anthropic 写进 Claude 模型卡). 
- **MINT: Evaluating LLMs in Multi-turn Interaction with Tools and Language Feedback** — Wang et al., ICLR 2024, arXiv:2309.10691. 多轮 + 自然语言反馈代表作, 研究 agent 如何利用反馈纠错. 

**五、MCP 时代 (新底座)**
- **MCP-Universe: Benchmarking LLMs with Real-World Model Context Protocol Servers** — Luo et al. (Salesforce), 2025, arXiv:2508.14704. MCP 评测时代地标 (11 个真实 MCP server、执行式评测, GPT-5 仅 43.72%). 
- **Model Context Protocol 规范本身** — Anthropic, 2024-11 发布. 不是论文, 但理解 2025 年后的评测必须读它的协议定义.

# 桥接 · 认知如何转化为方案

第二部分方案中的几乎每个关键决策, 都能追溯到第一部分的某条原理. 明确这条逻辑链, 是为了让方案不是凭经验拍脑袋, 而是有据可循、可被质疑和迭代. 

- **"有副作用工具必须 mock / 沙盒化"** ← 来自"可复现 vs 真实"困境与 StableToolBench 的教训 (§3、§2). 直接在真实环境对多模型反复跑会造成数据变更、不可复现、甚至事故. 
- **"为每条 case 配判定锚点, 而非只收集 prompt"** ← 来自不可能三角 (§2). 没有 ground-truth 就只能做 reference-free 的 LLM-Judge, 即三难图里最弱、最可被操纵的那条边. 
- **"规则优先, Judge 只用于开放式维度"** ← 来自匹配式/执行式"客观不可操纵"的优势与 LLM-Judge 的评分器偏向局限 (§2). 
- **"κ 校准作为 Judge 准入闸门"** ← 来自"LLM-Judge 可被操纵、有偏向"与"人工是黄金标准但不可规模化"的张力 (§2、§3). 校准是让二者对齐、使自动评测可信脱离人工的唯一路径. 
- **"不压成总分、保留分维度画像 + pass^k"** ← 来自诊断导向定位、分步诊断式方法 (§2) 、以及长程可靠性塌缩问题 (§3). 
- **"测试集要分层采样、补难例、版本化、切 holdout"** ← 来自选择偏差、污染与饱和、分布内 vs OOD (§3、§1⑦).

# 第二部分 · 垂域 Agent 测试集与自动化评测方案

## 1. 业务前提与定位

三个业务前提共同塑造了本方案的形态: 

**定位是诊断导向 (能力评估与改进方向), 而非门禁或签核.** 这意味着对单次跑分的极致复现性要求下降, 但对分维度、分意图类别的细粒度画像要求上升 —— 目标是回答"哪类任务、哪个能力弱", 而非给出 go/no-go 总分. 因此 bad case 库从附带产物上升为核心交付, 且可以容忍稍多的人工参与 (诊断是低频深度活动). 

**Agent 含副作用工具.** 这是方案中风险最高、必须最先处理的一环. 直接在真实环境对多个模型反复跑会造成真实数据变更、不可复现、甚至业务事故. 必须在 M0 插入工具副作用分级与 mock 层作为硬前置 (详见 §4). 

**现有 log 完整记录模型输入输出、工具调用、工具结果.** 这是好消息, 评测所需的核心可观测性已具备, M0 的 schema 工作大幅简化. 只需把现有字段整理成冻结的 trajectory schema, 并确认中间状态包含调用的时序与依赖关系 (评"调用效率""错误恢复"需要). 

## 2. 测试集构造

**核心思路**: 从业务真实调用 Agent 后落盘的 log 出发, 经清洗、脱敏、分层采样、分类、受控扩写、标注, 构造出带判定锚点的测试集. 从真实 log 出发的最大价值是分布保真度 —— 天然贴合真实用户意图分布, 这是公开基准最缺的. 

**流程**: log 清洗 → PII 脱敏 (必须是第一步) → 分层采样 → 意图分类 → 受控扩写 → A/B/C 三档标注. 

**五个风险与对策**: 
1.**只收集 prompt、没有判定锚点** (最大缺口). 一个能用于评测的测试集必须为每条 case 配判定依据, 否则只能做最弱的 reference-free Judge. 垂域 Agent 往往能定义可校验断言, 这是相对通用基准的优势, 务必用上. 
2.**选择偏差**. log 只含用户实际发过的 query, 不含"想问但知道做不到而没问的". 需主动补难例、边界例, 别只从 log 采样. 
3.**随机采样会被高频简单 query 淹没**. 应按意图类别 × 难度 × 频次 (头部/长尾) 分层采样, 长尾与易错例信号最高. 每类样本量要够 (50 条的成功率置信区间约 ±14%). 
4.**PII / 合规**. 真实 log 含隐私, 脱敏与数据治理必须是流水线第一步. 
5.**"扩写"要克制**. LLM 扩写有三重风险 —— 分布漂移、引入会被 Judge 偏好的人造痕迹、与候选模型同源造成污染. 扩写应服务于覆盖度 (同义改写鲁棒性、对抗变体), 并回测分布是否仍贴近真实, 而非单纯凑量. 

此外, 测试集要版本化、切出 holdout、定期刷新 —— 一旦团队拿它优化, 它就会饱和/被污染. 

**A/B/C 三档 ground-truth 策略**: 
- **A 档·可程序化断言** (最理想) : 直接写 checker, 断言期望调用的工具、关键参数取值、终态字段、final answer 的 golden 值或正则. 走纯规则, 零成本可复跑, 应尽量扩大占比. 对有副作用工具, A 档断言写的是**调用意图** (工具名 + 关键参数), 与 mock 层配套. 
- **B 档·rubric 评分**: 开放式但有评分标准, 写明各分档具体表现, 交给 LLM-Judge, 再人工抽检校准. 
- **C 档·仅参考轨迹**: 无法定义对错的, 标一条人工认可的 golden trajectory 作参照, 用 Judge 做相对比较. 

## 3. 评估维度体系

按"结果 → 工具 → 流程 → 质量安全 → 系统"五层组织. 垂域 Agent 应优先把能转成规则的维度规则化 (便宜、可复跑、无偏), LLM-Judge 只用于开放式维度, 人工只用于校准和高风险维度. 诊断导向下, 先选 6–8 个最有诊断价值的维度, 不必照搬全表. 

| 维度 | 层 | 实现方法 | 过程/结果 | 需 ground-truth |
|---|---|---|---|---|
| 任务完成度 / 成功率 | 结果 | 规则 (可校验断言) → 否则 Judge | 结果 | 是 |
| 最终答案正确性 | 结果 | 规则 (golden 字段比对) / Judge | 结果 | 是 |
| 相关性 / 是否答到点上 | 结果 | LLM-Judge | 结果 | 否 |
| 工具选择正确性 | 工具 | 规则 (vs golden tool) / Judge | 过程 | 是 |
| 参数填充正确性 | 工具 | 规则 (AST / 关键字段比对)  | 过程 | 是 |
| 调用有效性 (无幻觉工具/格式错)  | 工具 | 规则 | 过程 | 否 |
| 调用效率 (冗余调用、步数)  | 工具 | 规则 (计数 vs 期望步数)  | 过程 | 部分 |
| 指令 / 业务策略遵循 | 流程 | 硬约束规则 + 软约束 Judge | 过程 | 部分 |
| 多轮连贯性 / 上下文保持 | 流程 | LLM-Judge | 过程 | 否 |
| 澄清能力 (信息不足时反问)  | 流程 | LLM-Judge | 过程 | 否 |
| 错误恢复 (工具报错后纠正)  | 流程 | 规则 (检测 retry 成功) + Judge | 过程 | 否 |
| 安全 / 合规 (越权、敏感信息、注入抵抗)  | 质量安全 | 规则 (黑名单) + 人工 (高风险)  | 过程+结果 | 部分 |
| 幻觉 (编造事实/能力)  | 质量安全 | Judge + 规则 (grounding 校验)  | 结果 | 部分 |
| 话术 / 语气 / 风格 | 质量安全 | LLM-Judge | 结果 | 否 |
| 延迟 / Token 成本 / 异常率 | 系统 | 规则 / 指标采集 | — | 否 |

**两个关键设计点**: 不要把维度压成一个总分 —— 保留分维度 + 分意图类别的画像, 决策才有信号; 最终任务成功率额外做 **pass^k** (同一 case 跑 k 次的全成功率), 暴露长程可靠性塌缩, 这对垂域 Agent 上线尤其关键. 

## 4. 自动化评测 pipeline

**数据流**: 测试集 (含标注) + 候选模型 × N → 在 Agent 运行环境中产出 trajectory → 三类评测器并行处理 (规则评测器 / LLM-Judge / 人工抽检) → 聚合与校准 (bad case 抽取 + 一致性度量) → 输出四类产物 (评估明细、bad case 库、总结报告、auto-vs-human 一致性指标). 

**有副作用工具的分级与 mock (M0 硬前置)**. 给所有工具按副作用分三类处理: 

| 工具类型 | 例子 | 评测时如何处理 |
|---|---|---|
| 只读 | 查询、检索、计算 | 可直连真实环境, 或录制后回放 |
| 可逆写 | 草稿、可撤销操作 | 沙盒环境 + 测试账号, 跑后清理 |
| 不可逆/高风险写 | 下单、付款、发消息、删除 | 必须 mock: 拦截调用, 返回预设拟真结果, 断言"是否发起了正确调用"而非真执行 |

关键认知: 对不可逆写工具, **评测的判定点是"Agent 是否在正确时机、用正确参数发起了调用", 而非"调用是否真的执行成功"**. 这与 §2 的 A 档可程序化断言对应 —— 断言调用意图, mock 层拦截并返回拟真响应让 trajectory 继续往下走. 

**三类评测器**: 规则评测器读 trajectory 按 checker 输出对错; LLM-Judge 按 rubric 输出分数 + 理由; 人工抽检接口能调出 trajectory 给人打分. 

## 5. Judge 校准与人工闭环

这是决定自动评测能否可信脱离人工独立跑的核心环节. 

**Judge 工程注意三点**: judge 模型选用与候选模型**不同**的模型 (避免自我偏好); prompt 强制要求**先给理由后给分** (减少随意打分); 成对比较做**位置轮换** (消除位置偏置). 

**κ 校准 —— 核心闸门**. 逻辑链: 
1. 取一批"金标准"人工标注: 对 100–200 条 case 由标注者按判定契约打分作为 ground truth, 要分层覆盖各意图类别和难度, 不能只采简单例. 
2. 让 LLM-Judge 对同一批独立打分. 
3. 逐维度计算一致性: 分类型维度 (对/错) 用 Cohen's κ; 打分型维度 (1–5 分 rubric) 用加权 κ 或与人工分的相关系数. 
4. 按维度设准入: **只有 κ 达标的维度才允许 Judge 无人值守全量跑**; 不达标的维度, 要么回去改 rubric (往往是 rubric 模糊) 重测, 要么标记为"保留人工/规则", 不让 Judge 单独裁决. 

**κ 阈值怎么读**: 大致 0.6–0.8 不错、>0.8 很强、<0.4 基本不可用. 但阈值要按维度风险定 —— 安全合规类高风险维度应提到 0.8 甚至直接保留人工, 风格语气类低风险维度 0.5 可能就够诊断用. 

**人工闭环常态化**. 校准通过后人工不退场, 转为分层抽检 + 聚焦分歧: 把有限人力投在 Judge 低置信、auto-vs-human 历史分歧大、以及高风险维度的样本上, 持续监控 Judge 是否漂移. 这实现了"衡量自动化和人工一致性" —— 它是常态化质量监控, 而非一次性动作. 

## 6. 里程碑与执行路径

每阶段都设退出门, 门没过不进入下一阶段. 

**M0 · 对齐与前提 (1–2 周)**. 交付: 明确诊断导向定位、审计 log 完整性、**完成工具副作用分级表、不可逆工具的 mock 层就绪**、PII 治理方案、trajectory schema 冻结. 退出门: schema 冻结, 且三个业务前提都有明确落地方案, 尤其 mock 层可用. 

**M1 · 维度与 ground-truth 设计 (约 2 周)**. 四步: 维度裁剪与定级 (选 6–8 个, 能规则化的优先规则化) → 为每维度写"判定契约" (规则类写 checker 伪代码, Judge 类写带锚点的 rubric + 正反例) → 定义每条 case 的 A/B/C 档位并对接测试集构造 → 试标注 + 一致性验证 (抽 30–50 条, 2–3 人独立标注算 κ). 退出门: 判定契约定稿、标注规范定稿、试标注 κ 达标 (建议 >0.6). κ 不达标说明判定契约有歧义, 回去改契约而非责怪标注者. 

**M2 · 测试集 v0 (2–3 周)**. 交付: log 清洗 → PII 脱敏 → 分层采样 → 分类 → 受控扩写 → 按 A/B/C 三档标注. 退出门: 覆盖度与分布报告 (证明贴近真实且覆盖长尾) 、切出 holdout、各类样本量达统计功效. 

**M3 · Pipeline MVP + Judge 校准 (2–3 周)**. 三步: 搭三类评测器最小闭环 (小样本跑通) → κ 校准 (核心闸门, 逐维度算一致性定准入) → 固化人工抽检策略 (分层 + 聚焦分歧). 退出门: 关键维度 Judge κ > 各自阈值; 不达标维度有明确降级处理; 人工抽检分层策略上线. 

**M4 · 全量评测 + 报告 (1–2 周)**. 交付: 多模型在测试集上跑出 trajectory → 全量评测 → bad case 库 → 分维度/分类别报告 + pass^k. 退出门: 报告能直接支撑能力评估与改进方向的判断, 且经人工复核无系统性误判. 

**M5 · 固化与回归 (持续)**. 交付: 接入定期评测、测试集版本化与刷新机制、监控 Judge 漂移与数据污染. 退出门: 形成可重复运行、可追溯、可演进的常态化能力. 

**关键路径与最高风险点**: 依赖顺序为 M0 前提 → M1 维度与标注共设计 → M2 带标注测试集 → M3 校准 Judge → M4 全量 → M5 固化. 最容易被低估、最影响最终可信度的是 **M0 的工具 mock 层与 schema 冻结** (有副作用, 不做无法安全跑全量) 和 **M3 的 κ 校准** (决定自动评测能否脱离人工). 这两道门没过就往前冲, 后面产出的报告会因"测不到"或"评不准"而无法支撑决策.

## 附: 核心提醒

1. 本文引用的领域事实截至 2026 年初, 新基准与模型分数会快速变化. 
2. 方案中所有阈值 (κ >0.6、样本量、pass^k 的 k 值) 均为起点建议, 应按你的垂域风险偏好和实测调整. 
3. 最该先动手、风险最高的是有副作用工具的 mock 层 —— 它在关键路径上, 且不做就无法安全推进后续全部工作.
