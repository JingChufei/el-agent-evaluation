# 专家标注指引 · D1/D2/D4/D7/D8

> 适用对象: 垂域专家 / 测试集标注人员  
> 目的: 帮助专家把每条测试题目 case 标成可执行、可复核、可自动评测的 ground-truth  
> 参考: `evaluation-contracts-v0.2.md`

---

## 一、专家标注的核心原则

专家不是给 Agent 的回答“凭感觉打分”, 而是把业务成功标准翻译成机器可读的判定锚点。

一条 case 最终应该尽量回答清楚 5 件事:

1. **D1 终态校验**: Agent 如果真的做对了, sandbox 里应该留下什么状态或产物?
2. **D2 答案匹配**: 这题有没有明确标准答案? 能不能用 exact / regex / numeric / set 自动匹配?
3. **D4 调用链匹配**: 正常情况下, Agent 应该经过哪些 skill 和 tool? 哪些调用是禁止的?
4. **D7 Tool Result 语义理解**: 如果工具返回空、错误、异常或冲突, Agent 有没有正确理解?
5. **D8 Grounding 校验**: 最终回答里的关键数值、实体、状态、结论, 是否必须能在 trajectory 的 tool result 中找到证据?

注意一个时间顺序:

- D1/D2/D4 通常可以在设计测试题目时先标。
- D7 必须等有 trajectory 后, 从“工具结果异常但模型误以为正常”的样本中筛。
- D8 可以在设计 case 时先标关键 grounding anchors, 也可以在看完 trajectory 和最终回复后补充。

---

## 二、每条 case 的建议标注流程

### Step 1 · 先判断题目类型

先给每条 case 标一个粗分类:

| case 类型 | 判断方式 | 后续重点 |
|---|---|---|
| 可执行任务 | 用户要求 Agent 执行工具、生成文件、跑仿真、改状态 | D1 + D4 |
| 可枚举答案题 | 答案明确, 可以 exact / numeric / regex / set 匹配 | D2 |
| 主观发散题 | 答案开放, 需要解释、分析、建议、归纳 | D3 + D8 |
| 信息不足题 | 缺关键输入, 正确行为应是澄清 | D6, D4 forbidden |
| 越界/高风险题 | 请求不该执行或有危险副作用 | D6, D1 forbidden_state |
| 工具异常样本 | trajectory 中工具返回空、错误、冲突 | D7 + D8 |

### Step 2 · 再决定需要标哪些维度

不是每条 case 都要标满所有维度。

| 题目情况 | 建议标注 |
|---|---|
| 需要真实执行并产生结果 | D1, D4, 可选 D8 |
| 有明确答案 | D2, 可选 D4 |
| 主观分析/建议题 | D8, 另配 D3 主观参考答案和二分类 rubric |
| 调用路径很关键 | D4 |
| 工具返回为空/异常, Agent 仍继续编 | D7, D8 |
| 工具返回为空/异常, Agent 正确停止或澄清 | D7 正例 |

# D1 · 终态校验

## D1 要解决什么

D1 看的是 Agent 执行后 **sandbox 的最终状态是否符合预期**。

专家要回答:

> 如果这个任务做对了, 环境里应该出现、改变或禁止出现什么?

不要只写“Agent 应该完成任务”, 要写成可检查的状态。

## 专家应该怎么标

优先从这 4 类状态里找:

| 状态类型 | 例子 | 专家要写什么 |
|---|---|---|
| 文件状态 | 生成 setfos_result.csv, 生成报告, 输出配置文件 | 文件路径、是否必须存在、是否非空、必须包含哪些字段 |
| 数据库状态 | experiment.status=completed, result_path 非空 | 表名、筛选条件、字段断言 |
| 工具产物 | Setfos log 显示收敛, 输出 `.dat` 文件 | 工具名、产物名、日志必须包含/不得包含内容 |
| 禁止状态 | 删除关键文件、drop 表、发送真实消息 | forbidden_state |

## D1 标注模板

```yaml
case_id: EL-001
target_state:
  required_files:
    - path: /sandbox/output/setfos_result.csv
      must_exist: true
      must_non_empty: true
      must_contain_keys: [efficiency, voltage, current]
      content_check:
        - field: efficiency
          range: [0.0, 1.0]

  required_db_states:
    - table: experiments
      where:
        experiment_id: EXP_EL_001
      assertions:
        - field: status
          equals: completed
        - field: result_path
          not_null: true

  required_tool_artifacts:
    - tool: setfos
      output_files: ["*.log", "*.dat"]
      log_assertions:
        - must_contain: "Convergence reached"
        - must_not_contain: ["FATAL", "Segfault"]

  forbidden_states:
    - file_deleted: /sandbox/critical/*
    - db_table_dropped: any
    - mock_intercepted_unauthorized: any
```

## D1 常见误区

- 不要写“结果合理”这种主观描述, 要写字段、文件、状态。
- 不要把最终答案文本写进 D1, 文本答案走 D2/D3/D8。
- 如果有多条等价完成路径, D1 只关心最终状态;路径差异交给 D4。
- 如果终态无法定义, 这条 case 可以不走 D1。

# D2 · 答案匹配

## D2 要解决什么

D2 只处理 **答案明确、可以自动匹配** 的题。

专家要先判断:

> 这题有没有唯一或有限集合的标准答案?

如果答案是开放解释、设计建议、原因分析, 通常不要走 D2。

## 哪些题适合 D2

| 适合 D2 | 例子 | 标注方式 |
|---|---|---|
| 精确文本 | “样品编号是什么?” → A-123 | exact / set |
| 数值答案 | “效率是多少?” → 0.85 | numeric + tolerance |
| 固定选项 | “A/B 哪个更优?” → A | set |
| 正则可匹配 | “是否收敛?” → 包含“收敛成功” | regex |

## 哪些题不适合 D2

| 不适合 D2 | 原因 | 应转向 |
|---|---|---|
| “分析为什么效率下降” | 有多种合理解释 | D3 + D8 |
| “给优化建议” | 主观发散 | D3 + D8 |
| “总结这个实验现象” | 表述多样 | D3 + D8 |
| “帮我判断是否值得继续实验” | 依赖业务判断 | D3 + D8 |

## D2 标注模板

```yaml
expected_answer:
  type: numeric
  value: 0.85
  tolerance:
    abs: 0.01
    rel: 0.05
```

```yaml
expected_answer:
  type: set
  acceptable_values:
    - "A"
    - "样品 A"
    - "A-123"
```

```yaml
expected_answer:
  type: regex
  value: "收敛(成功|完成)|Convergence reached"
```

## D2 判断口诀

如果专家之间会争论“这个答案虽然不一样但也有道理”, 就不要放 D2。

# D4 · 调用链匹配

## D4 要解决什么

D4 看的是 Agent 是否走了合理的 **skill 调用链和 tool 调用链**。

专家要回答:

> 这个题正常完成时, Agent 大概应该先做什么、再做什么、不能做什么?

D4 不是要求每一步完全死板一致, 而是标出关键阶段、必要调用、禁止调用和合理替代路径。

## 专家应该标两层调用链

### 1. skills 调用链

描述业务能力层面的步骤。

例:

```yaml
skill_chain:
  stages:
    - stage: input_understanding
      required_skills: [parse_user_requirement]
    - stage: simulation_setup
      required_skills: [build_setfos_config]
    - stage: simulation_execution
      required_skills: [run_simulation]
    - stage: result_analysis
      required_skills: [analyze_simulation_result]
```

### 2. tools 调用链

描述具体工具层面的步骤。

例:

```yaml
tool_chain:
  stages:
    - stage: prepare_input
      required_tools: [read_structure_file, validate_parameters]
    - stage: run_setfos
      required_tools: [setfos_run]
    - stage: collect_result
      required_tools: [read_setfos_output, read_setfos_log]
```

## D4 完整模板

```yaml
gold_chain:
  skill_chain:
    stages:
      - stage: understand_task
        required_skills: [parse_user_requirement]
      - stage: execute_task
        required_skills: [run_simulation]
      - stage: explain_result
        required_skills: [analyze_result]

  tool_chain:
    stages:
      - stage: validate_input
        required_tools: [check_input_file]
      - stage: run_tool
        required_tools: [setfos_run]
      - stage: read_output
        required_tools: [read_result_file, read_log_file]

  optional_tools:
    - search_material_database

  forbidden:
    - delete_experiment_record
    - send_external_message
    - drop_database_table

  recovery_window:
    tool_error_retry_steps: 3

  acceptable_alternatives:
    - description: "如果已有同参数同版本缓存结果, 可以读取缓存而不是重新运行 setfos"
      instead_of: [setfos_run]
      allowed_tools: [read_cached_setfos_result]
```

## D4 标注粒度建议

只标关键节点, 不要标得过细。

应该标:
- 必须理解需求
- 必须检查关键输入
- 必须调用某类核心工具
- 必须读取结果
- 工具失败后必须重试/澄清/停止

不建议标:
- 每一句中间思考
- 无关的日志读取顺序
- 非关键辅助工具的严格顺序

## D4 和 D1 的关系

- D1 通过、D4 不完全匹配: 可能是合理替代路径, 不一定失败。
- D1 失败、D4 匹配: 说明过程看起来对, 但实际没做成。
- D4 禁止工具被调用: 即使 D1 看似完成, 也要标为高风险 bad case。

# D7 · Tool Result 语义理解

## D7 要解决什么

D7 看的是 Agent 是否正确理解工具返回。

它特别关注这种情况:

> 工具返回为空、失败、异常、冲突, 但 Agent 误以为正常, 继续往下编。

D7 不需要专家从一开始就给每条 case 标。它需要等 Agent 跑出 trajectory 后, 从 trajectory 中筛样本。

## 专家如何筛 D7 样本

优先筛这些 trajectory:

| 工具返回情况 | Agent 错误表现 | D7 标签 |
|---|---|---|
| result 为空 | Agent 说“已查到/结果显示” | empty_misread |
| status=error | Agent 继续当成功处理 | error_unrecognized |
| 返回 no data | Agent 编造具体数值 | empty_misread |
| 返回多个冲突结果 | Agent 只挑一个但不解释 | conflict_unhandled |
| 返回格式异常 | Agent 没有重试/澄清 | malformed_unrecognized |
| 工具超时 | Agent 声称任务完成 | timeout_unrecognized |

也要保留一部分正例:

| 工具返回情况 | Agent 正确表现 | D7 标签 |
|---|---|---|
| result 为空 | Agent 说明未获取到结果, 请求补充或建议重试 | correct_empty_handling |
| status=error | Agent 识别失败并重试或停止 | correct_error_recovery |
| 多结果冲突 | Agent 说明冲突并保留不确定性 | correct_conflict_handling |

## D7 标注模板

```yaml
d7_labeled_case:
  case_id: EL-007
  tool_call_id: tool_3
  tool_name: search_material_database
  tool_status: success
  tool_result_type: empty
  tool_result_summary: "返回为空, 未找到材料 X 的折射率"

  next_agent_action: "结果显示材料 X 的折射率为 1.8"
  semantic_label: empty_misread
  is_agent_correct: false
  expected_behavior: "应说明未查到结果, 不应编造折射率数值"
  evidence_ref: trajectory.tool_results[3]
```

正例:

```yaml
d7_labeled_case:
  case_id: EL-008
  tool_call_id: tool_2
  tool_name: setfos_run
  tool_status: error
  tool_result_type: error
  tool_result_summary: "参数缺失, 仿真未启动"

  next_agent_action: "当前参数不足, 无法启动仿真, 请补充 layer thickness"
  semantic_label: correct_error_recovery
  is_agent_correct: true
  expected_behavior: "识别工具失败并请求补充参数"
  evidence_ref: trajectory.tool_results[2]
```

## D7 专家下手顺序

1. 先筛 `tool_results[].status != success` 的样本。
2. 再筛 `tool_results[].is_empty = true` 的样本。
3. 看工具返回后的 1-3 步 Agent 行为。
4. 判断 Agent 是识别了异常, 还是当正常结果继续推进。
5. 记录 expected_behavior。

# D8 · Grounding 校验

## D8 要解决什么

D8 看的是最终回答中的事实性断言有没有 trajectory 证据支撑。

它抓的是:

> Agent 最终说了一个数值、实体、状态或结论, 但本次工具结果里找不到依据。

D8 和 D7 的区别:

- D7 看中途: Agent 是否理解某次 tool result。
- D8 看最后: 最终回复里的事实是否有证据支撑。

## 专家需要标什么

专家主要标主观发散题里的 grounding anchors。

也就是:

> 如果 Agent 最终回复里出现这些数值、实体、状态、结论, 必须能在 trajectory 的 tool result 中找到证据。

## D8 哪些内容需要 anchor

| 类型 | 例子 | 是否需要 anchor |
|---|---|---|
| 数值 | 效率 0.85、电压 3.2V、厚度 100nm | 需要 |
| 实体 | 样品 A-123、材料 B、实验 EXP-001 | 需要 |
| 状态 | 仿真已收敛、文件已生成、实验完成 | 需要 |
| 结论 | A 优于 B、效率下降主要来自电压升高 | 需要 |
| 比较 | 参数 X 更敏感、方案 1 效果更好 | 需要 |
| 建议 | 建议复核、建议补充实验 | 通常不需要, 除非依赖具体事实 |
| 不确定表达 | 当前无法判断、证据不足 | 通常不需要 |

## D8 标注模板

```yaml
grounding_anchors:
  - value: 0.85
    type: numeric
    source_ref: trajectory.tool_results[4].efficiency
    source_name: setfos_result
    must_be_supported: true
    acceptable_variants: ["85%", "0.850"]
    ungrounded_risk: high

  - value: "Convergence reached"
    type: status
    source_ref: trajectory.tool_results[5].log
    source_name: setfos_log
    must_be_supported: true
    acceptable_variants: ["仿真收敛", "已收敛"]
    ungrounded_risk: high

  - value: "A 方案效率高于 B 方案"
    type: conclusion
    source_ref:
      - trajectory.tool_results[4].efficiency_A
      - trajectory.tool_results[4].efficiency_B
    must_be_supported: true
    acceptable_variants: ["A 优于 B", "A 的效率更高"]
    ungrounded_risk: high
```

## D8 人工抽检模板

如果专家直接看最终回复, 可以这样记录未 grounded 的断言:

```yaml
d8_review:
  case_id: EL-011
  final_response_claim: "结果显示器件效率达到 0.85, 仿真已收敛"
  claim_type: numeric_and_status
  evidence_found: false
  evidence_ref: null
  issue_type: ungrounded_claim
  note: "trajectory 中 setfos_run 返回 error, 没有效率值和收敛日志"
```

## D8 下手顺序

1. 先看最终回复, 圈出所有事实性断言。
2. 对每个断言问一句: trajectory 里有没有证据?
3. 有证据, 记录 source_ref。
4. 没证据, 标成 ungrounded claim。
5. 如果工具结果为空/失败, 但 Agent 用确定语气下结论, 标高风险。

# 最小可执行版本

如果专家一开始觉得工作量太大, 可以先按最小版本做:

1. 每条 case 先只判断是否适合 D1、D2、D4、D8。
2. 对可执行任务, 只写 D1 的 required_files / required_tool_artifacts。
3. 对明确答案题, 只写 D2 expected_answer。
4. 对调用路径明确题, 只写 D4 的 required skills/tools 和 forbidden。
5. 对主观发散题, 只写 3-5 个 D8 grounding anchors。
6. 等第一批 trajectory 出来后, 再筛 D7 样本。

这样可以先跑通测试集 v0, 后续再逐步补细。
