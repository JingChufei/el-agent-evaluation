#### 1. EL Agent 评估维度 & 判定契约确定

本周完成当前业务下 8 个评估维度的判定契约收敛, 文档按统一模板描述每个维度的评测对象、输入、ground-truth 需求、输出指标、实现方法、判定规则、边界条件、校准要求和失败模式。

当前维度体系:

| 维度 | 名称 | 定位 | 主路径 | 标注需求 |
| --- | --- | --- | --- | --- |
| D1 | 终态校验 | 结果层 | sandbox 快照 diff + 规则校验 | per-case target_state |
| D2 | 答案匹配 | 结果层 | 规则匹配 | per-case expected_answer |
| D3 | 答案质量 | 结果层 | LLM-Judge + 二分类 rubric | 主观参考答案 + 二分类 rubric |
| D4 | 调用链匹配 | 过程层·宏观 | 规则匹配 | per-case gold_chain |
| D5 | Tool Call 格式合法性 | 过程层·微观 | 规则校验 | 无 |
| D6 | 调用决策合理性 | 过程层·微观 | 规则 + 轻 Judge | 负样本子集 |
| D7 | Tool Result 语义理解 | 过程层·微观 | 规则 + Judge 兜底 | 子集人工标注 |
| D8 | Grounding 校验 | 质量层 | 规则 + NLI 兜底 | 可选 grounding anchors |

关键收敛点:

- D1 终态校验从 mock 调用意图核对升级为 sandbox 真实执行后的状态快照 diff。判定对象从"Agent 想做什么"转为"Agent 实际做成了什么", 这会成为结果层主判维度。
- D5 作为零 ground-truth 的基础格式检查最先执行, 用于提前暴露工具名、参数 JSON、schema、幻觉工具等低层错误。
- D4 与 D1 联动处理"合理替代路径": 如果调用链不完全匹配 gold_chain, 但 D1 终态通过, 不直接判失败, 而标记为合理替代路径。
- D6 单独覆盖负样本和调用决策问题, 避免 Agent 在信息不足、越界、不需要工具时仍然乱调用工具。
- D7 和 D8 分别覆盖"中途歪了"和"最后编了": D7 看 Agent 是否正确理解工具返回, D8 看最终回复是否有 trajectory 证据支撑。
- D3 答案质量从 1-5 分制改为二分类对/错判定。具体逻辑为: 测试题目 → 专家写标准主观答案 / SOTA 模型生成高质量主观答案后专家审核 → 专家手写 rubric 或 LLM 抽取 rubric 后专家审核 → Judge 按二分类 rubric 判定通过/不通过。

D3 的这次调整比较关键。原先 1-5 分 rubric 容易带来分档边界不稳定、Judge 打分漂移和解释成本高的问题。改成二分类后, 更适合当前"诊断与回归"场景: case 级只判断是否满足该题的关键质量要求, 聚合时看答案质量通过率和失败原因分布。D8 grounding 严重失败时, D3 不再做分数封顶, 而是直接判为不通过。

当前契约还同步明确了 trajectory schema 的字段需求, 包括 `user_query`, `final_response`, `tool_calls[]`, `tool_results[].status`, `tool_results[].is_empty`, `sandbox_initial_snapshot_ref`, `sandbox_final_snapshot_ref`, `sandbox_intercept_log[]` 等。后续开发侧 schema 落地会直接影响 D1/D4/D5/D7/D8 是否可自动化评测。

#### 2. 团队配合任务需求厘清

本周基于判定契约和执行路径, 将跨团队任务拆成四类协作输入, 并按团队形成独立的协作对齐草案。由于当前项目负责人尚未最终确定, 文档措辞已从"OKR/目标下达"调整为"协作对齐草案/可讨论事项/可能沉淀的材料", 避免给其他团队造成越权或任务指派的感觉。

四类协作草案:

| 协作对象 | 重点任务 | 核心接口产出 |
| --- | --- | --- |
| 评测侧 | 契约维护、checker 实现、Judge 校准、pipeline MVP | 评测契约、checker、评测结果 schema、bad case 报告 |
| 开发团队 | sandbox、trajectory schema、Sandbox API/SDK | snapshot schema、trajectory schema、sandbox 调度接口 |
| 专家团队 | 测试集 ground-truth、标注规范、人工金标准 | target_state、expected_answer、gold_chain、二分类 rubric、人工金标准集 |
| Agent 团队 | skills 行为契约、意图分类法、业务难例 | skills spec、意图分类法、skill-tool 依赖、负样本/难例 |

当前最关键的三类接口:

1. **sandbox 接口**: 开发侧需要明确 create/reset/snapshot/destroy 等调度能力, 以及文件、数据库、工具产物、mock 拦截记录的 snapshot 格式。D1 终态校验是否能落地, 主要依赖这个接口。
2. **测试集判定规则接口**: 专家侧需要为不同类型 case 标注可执行 ground-truth。特别是 D1 的 `target_state`、D4 的 `gold_chain`、D3 的主观参考答案 + 二分类 rubric, 会直接决定 checker 和 Judge 是否能稳定工作。
3. **skills 行为契约接口**: Agent 团队需要把每个 skill 的用途、输入输出、工具依赖、副作用、边界条件讲清楚。专家团队据此做意图分类和测试集分层, 开发团队据此做工具副作用分级和 sandbox 化。

关于"意图分类法", 本周也进一步明确其含义: 它不是管理分类, 而是把用户 query 按业务任务类型划分, 例如仿真执行、结果分析、参数查询、设计建议、高风险操作、out_of_scope 等。它的作用是支撑测试集分层、报告切片和 bad case 归因。最终报告不应只给总通过率, 而要能回答"哪类意图弱、对应哪个 skill 或工具链路弱"。

#### 阶段性结论

当前项目已经从"评测方案设计"推进到"判定契约 + 团队接口定义"阶段。下一阶段的主要风险不再是方法论是否成立, 而是接口是否能真正落地:

- sandbox 是否能稳定复现, 决定 D1 能否作为结果层主判;
- trajectory schema 是否覆盖必要字段, 决定过程层维度能否自动化;
- 专家 ground-truth 是否可执行、可复核, 决定测试集是否能从 query 集合变成评测集;
- D3/D6/D7/D8 的 Judge κ 是否达标, 决定开放式维度能否低成本规模化运行。
